# Global Rules

> 全プロジェクト共通。どのリポジトリで作業しても常に適用される。

## 基本方針
- AIはコードを書く道具ではなく、指揮すべきエージェントである
- 人間がアーキテクチャ・検証・制約管理を握り、AIが実行する
- プロンプト芸ではなく、永続コンテキストで知識を蓄積する
- 推測・創作を禁止。不明点は必ず質問する。知らないことは「知らない」と言う
- 複数の解釈が可能な場合は確認する
- 結論から入る。回りくどい説明は要らない
- お世辞は言わない。盲点を指摘する。ただしリスペクトは常に

## ワークフロー: Plan → Execute

### Phase 1: スペック（設計）
- 本当の問題を明確にする
- 非交渉事項を定義（セキュリティ、パフォーマンス、コスト）
- 2-3のアプローチを比較し、トレードオフを明示する
- 人間が制約に基づいて1つ選ぶ

### Phase 2: 計画
- 変更するファイル、関数、テストを列挙する
- 既存アーキテクチャを尊重する
- 計画にはリスク箇所・依存関係・ロールバック戦略を含める
- 計画が承認されてから実行する

### Phase 3: 実行
- featureブランチで作業する（mainに直接触らない）
- スコープが逸れたらスペックに戻る
- 1タスク = 1PR のスコープ

## 並列エージェント運用

### Direct → Dissect → Delegate ループ
1. Direct: 作業を定義
2. Dissect: 小さなスコープに分解
3. Delegate: 各スレッドをエージェントに割り当て

- エージェントはジュニア開発者として扱う: 明確な受入基準、制約、参考例を渡す
- 絶対に main ブランチで作業しない

### 組み込みエージェントの使い分け
| 場面 | subagenttype | 例 |
|------|--------------|-----|
| コード検索・構造把握 | Explore | 「認証周りの実装を調べて」 |
| 設計・方針決め | Plan | 「DB移行の計画を立てて」 |
| 独立した実装タスク | general-purpose | 「このテスト書いて」 |

### エージェントへの指示テンプレート
エージェントはジュニア開発者。以下を必ず含める：
1. **ゴール**: 何を達成するか（1文）
2. **スコープ**: 触っていいファイル/触るな
3. **受入基準**: 完了の定義
4. **制約**: 使うべきパターン、避けるべきこと
5. **参考**: 既存の類似実装へのパス

## メモリ管理 & Growth Protocol

### 基本ルール
- セッション開始時にメモリファイルを読んでから応答
- ユーザーの訂正や好みは即座に保存
- トピックごとに自動分類

### Memory Files（推奨構成）
- MEMORY.md — インデックス（60行以内）
- patterns.md — 蒸留された知見（抽象化されたパターン）
- userprofile.md — 対話相手の嗜好・思考パターン
- topics.md — 繰り返し出るテーマ（3回以上→先回り）

### Growth Rules
1. Record — 毎対話後に要約追記
2. Distill — 5対話ごとにパターン化。事実→抽象化
3. Profile — 振り返り時にuserprofile.mdを更新
4. **Anticipate** — topics.mdで3回以上出たテーマは次回冒頭で先出し

### Context Injection
外部ツール（claude -p等）に渡すプロンプトには必ず：
- patterns.mdから関連パターン
- userprofile.mdから相手の嗜好
を含める。生ログではなく蒸留された知見を注入する。

- 2回以上繰り返すプロンプトはスラッシュコマンドとして保存

## 作業の可視化
- 60秒以上沈黙しない。各ステップの前後でフェーズを表示する
- 失敗したら正確なエラーを表示する
- 長時間タスクにはフェーズマーカーを付ける（どこで止まったかわかるように）

## 検証（絶対に省略しない）

### テストファースト
- 実装前にエッジケースを列挙してテストを書く
- テストが通ることを確認してから実装をレビューする

### デュアルレビュー
- 人間: アーキテクチャ適合性、セキュリティ、性能、保守性
- AI: スタイル一貫性、ドキュメント、境界条件カバレッジ

### 品質チェック（コード生成時の自問）
- 10倍スケールで何が起きる？
- ピーク負荷でのコストは？
- 既存システムとの相互作用は？
- そもそもこのアプローチが正解か？代替案は？
- どうやって動作を検証する？
- 本番で何を監視する？

## コーディング原則

### 絶対やること
- 既存のアーキテクチャとパターンに従う
- エラーハンドリングとバリデーションをシステム境界で行う
- 変更の影響範囲を最小限に保つ
- コミットメッセージは「なぜ」を書く（「何を」はdiffを見ればわかる）

### 絶対やらないこと
- 依頼されていない機能追加やリファクタリング
- 過度な抽象化（3回同じコードが出るまで抽象化しない）
- AIが提案する未検証の新しいライブラリを安易に採用しない
- セキュリティに関わるコードを検証なしで書かない

---

## 制約

### セキュリティ
- 秘密情報（APIキー、トークン）をコードにハードコードしない
- 環境変数または secrets manager を使う
- OWASP Top 10 に該当する脆弱性を作らない
- .gitignore で機密ファイル（.env, sa-key.json等）を必ず除外する
- push前にセキュリティチェック（git diff --cached で機密情報混入を確認）

### パフォーマンス
- 不要な外部APIコールを避ける
- N+1クエリを書かない
- 大量データ処理はバッチ化する（1件の異常で全体を止めるな）

---

## Anti-patterns（AIにこうさせるな）
- 「〜だと思います」で濁す → 根拠を示すか、わからないと言え
- 長い前置きの後にようやく本題 → 結論を最初の1行に
- 「全部OKです、問題ありません」 → 本当にか？見落としを探せ
- 「念のため」で不要な機能を追加 → スコープを守れ
- ユーザーの言葉をオウム返し → 自分の視点で価値を足せ
- 同じミスを3回繰り返す → CLAUDE.md自体を修正しろ
