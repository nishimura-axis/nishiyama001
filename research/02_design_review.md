## ガガロットAI全文を踏まえた改善点

### 1. コンテンツパイプライン → 「プロジェクトパイプライン」に読み替え
ガガロットのコンテンツパイプライン（アイデア→スクリプト→撮影→編集→公開）は俺たちには不要。
だが「パイプライン的な可視化」自体は有用。
→ AXAD Phase 2の8ステップをパイプライン表示するのはアリ。
→ ただしMVPには入れない。タスクボードで十分。

### 2. カレンダー機能
ガガロットが「最も重要なコンポーネント」と主張。理由は「AIがスケジュールを忘れる」問題。
→ OpenClawのCron jobsが既にこの役割を担っている。
→ ダッシュボードにCronジョブのビューを追加するだけで済む。
→ Phase 2で入れる。MVPには不要。

### 3. Convex vs 俺たちの構成
ガガロットはConvexのリアクティブクエリに依存。俺たちはOpenClaw Gateway API + GitHub API。
→ Convexの方がリアルタイム性は高い（DB変更が即UI反映）
→ ただし俺たちは既にOpenClawインフラがある。新たにConvexを入れるメリットが薄い
→ SSE + SWRの設計で問題ない

### 4. 「AIに作らせる」前提 vs 設計→実装の分業
ガガロットは「OpenClawに指示して作らせる」スタイル。
俺たちは西山1号が設計 → 西山2号が実装する分業。
→ これは俺たちの方が品質管理しやすい。にっしーのレビューも入る。
→ ガガロットのプロンプト例は2号のタスク定義に使える参考資料として残す。

## 要件定義の穴・改善点

### A. VercelからGateway接続の具体策が未記載
**問題:** VercelにデプロイしたNext.jsアプリから、にっしーのMac miniで動くGateway（localhost:18789）にどう接続するか。
**選択肢:**
1. Tailscale経由（Mac miniのTailscale IPでアクセス）
2. Cloudflare Tunnel
3. ダッシュボード自体もMac miniでローカル起動（Vercel使わない）
4. Next.jsのAPI RouteはMac miniローカルで動かし、フロントだけVercel

→ **選択肢3が最もシンプル。** MVPではローカル起動でいい。Vercelデプロイは後から考える。
→ 質問①の回答次第で決定。

### B. 「思考ログ」のデータソースが曖昧
**問題:** thinking_log/*.md を60秒pollingで読むとあるが、影分身がこのファイルを書く規約がない。
**改善:** 
- `memory/YYYY-MM-DD.md` の作業ログ（🕐形式）を思考ログとして使う
- Telegramの送信メッセージ（セッション履歴）も併用
- thinking_log/ は廃止してシンプルにする

### C. tasks.jsonの更新タイミングが未定義
**問題:** 各影分身がいつtasks.jsonを更新するか書いていない。
**改善:** CLAUDE.mdに「タスク開始時・完了時にtasks.jsonを更新すること」を明記する。

### D. 「回答フロー」の具体性不足
**問題:** US-4「未回答の質問一覧を見て、すぐに回答できるようにしたい」とあるが、回答はどうやって影分身に届くのか。
**改善案:**
1. ダッシュボードで回答 → questions/*.mdのstatusをansweredに更新 + 回答本文を追記
2. 影分身が次回起動時にquestions/を読み、answeredのものを確認
3. または回答をTelegram経由で影分身に直接送る（既存のフロー）
→ MVPではTelegram経由でいい。ダッシュボードからの回答機能はPhase 2。

### E. 認証が甘い
**問題:** 「初期: なし」は危険。ローカルネットワークでもBasic Auth程度は入れるべき。
**改善:** MVPでもBasic Auth必須。環境変数で設定。
