# patterns.md — 蒸留された知見

## にっしーとの壁打ち
- 結論→構造→詳細の順で出すと判断が速い
- 「未確認事項」を明示すると信頼される。推測で埋めると嫌がる
- MTG要約は「決定事項・宿題・次回日程」の3点セットが必須

## タスク管理
- にっしー依存タスクが滞留しやすい。催促より「次に聞かれそうなこと」を先に準備しておく方が有効
- tasks.jsonの更新漏れが頻発（task-004は3/6完了なのに3/14まで未更新）→ 対話で完了を検知したら即更新

## チーム連携
- 3号への完了報告は「にっしー＋5号」同時送信が必須（3/14の品質チェック漏れ教訓）
- 7号（太一用）はセットアップ済みだが利用実績ゼロ。定期確認を継続

## 設計方針
- マルチテナント設計時は「データ混在させない」が鉄則（デマンド・リバティ案件）
- 外部連携（アドアシ等）はDB差異をコンバーターで吸収、認証は相手側形式に統一

## self-work-check
- にっしー依存で全タスクblocked時でも、メモリ整理・パターン蒸留・日次ログ品質改善は自律でできる
- 同じHEARTBEAT_OKを10回以上繰り返すのは無駄。自律作業を探せ
