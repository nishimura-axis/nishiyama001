# 西山7号（DProリサーチ分身）設計書 v0.1

## 1. ミッション定義

**一文定義:** DPro MCPを使い、運用者の市場リサーチを自動化し、売上拡大に直結するクリエイティブ・競合インサイトを配信する専任エージェント。

**Phase 1ゴール:** 太一が「これは使える」と言い、自発的に使い続けている状態。
**成功基準:** 太一の売上に繋がるアクション（新CR制作、出稿判断、競合対策）が1つでも生まれたか。

---

## 2. 自動リサーチ設計

### 太一への配信メニュー（Phase 1）

PoCで実証済みの「一括取得→分析→配信」パターンを、太一の日常業務に埋め込む。

#### A. デイリーモニタリング（毎朝9:00配信）
**目的:** 太一が担当する商材/ジャンルの新着CRを毎朝キャッチアップ
- 太一の担当商材で過去24時間の新着CR動画を取得（get_items, ソート=新着）
- 媒体: 太一が主に出稿する媒体に絞る
- 配信内容:
  - 新着CR数
  - 注目CR（imp/再生数が急伸しているもの）のサマリー
  - 競合代理店の動き（新規出稿があれば通知）
- **所要時間:** 太一が朝1分で状況把握できる粒度

#### B. 週次競合レポート（毎週月曜9:00配信）
**目的:** 週単位のトレンド変化を定点観測
- 太一の担当ジャンルTOP100を取得（get_items, ソート=imp順）
- 先週比の変動を分析:
  - 新規ランクインCR
  - 代理店シェアの変動
  - メニュー別・訴求パターン別の構成比変化
- **PoC再現:** 2/25 TCB分析と同じフォーマット

#### C. オンデマンド分析（太一がTelegramで依頼）
**目的:** 「あの商材のCR見たい」を即座に返す
- 太一がTelegramで自然言語で依頼
  - 例:「TCBのTikTok直近1ヶ月TOP50出して」
  - 例:「ゴリラクリニックの訴求パターン分析して」
  - 例:「この記事LPの内容教えて」（URL指定→read_transition_url_text_content）
- 7号がDPro MCPを叩いて即レスポンス

### 導線設計（太一の1日）

```
09:00  Telegram通知 → デイリーモニタリング確認（1分）
       → 気になるCRがあれば「詳しく」と返信 → 7号が詳細分析
日中   思いついた時にTelegramで分析依頼 → 即返答
月曜   週次レポートで競合動向を俯瞰 → 出稿戦略に反映
```

**設計思想:** 太一から7号に聞きに行くのではなく、7号から太一に必要な情報をpushする。太一は受け取って判断するだけ。

## 3. 技術構成

### アーキテクチャ

```
太一(Telegram) <-> OpenClaw Gateway <-> 西山7号(Agent)
                                            |
                                      DPro MCP Server
                                            |
                                      DPro API (OAuth2)
```

### Mac mini上の実行方法

- **エージェント:** OpenClawのagentとして登録（CLIコマンド経由）
- **定期実行:** OpenClaw CRONで毎朝9:00 / 毎週月曜にリサーチジョブ起動
- **オンデマンド:** Telegram DMで太一が直接対話
- **workspace:** ~/Nishiyama007Vault/ （分析結果の蓄積用）

### DPro MCP認証の解決策（最大リスク）

OAuth2のブラウザ認証をMac mini無人環境で維持する問題。3パターン検討：

#### パターンA: リフレッシュトークン長期保持（推奨）
- 初回のみブラウザでOAuth2認証 → アクセストークン + リフレッシュトークン取得
- リフレッシュトークンで自動更新するスクリプトをpm2常駐
- **前提:** DPro APIがリフレッシュトークンを発行すること
- **リスク:** リフレッシュトークンの有効期限が短い場合は使えない

#### パターンB: ブラウザ自動認証（Playwright）
- Playwrightでブラウザ認証を自動化
- CRONでトークン期限前に再認証を走らせる
- **メリット:** 確実にトークン取得できる
- **リスク:** DProのログインUIが変わると壊れる

#### パターンC: トークン手動更新運用
- トークン切れたら太一 or にっしーが手動で再認証
- 7号がトークン切れを検知したらTelegramで通知
- **メリット:** 実装が最もシンプル
- **リスク:** 手動介入が必要。太一の負担になる

**→ 判断にはQ4の回答が必要**

## 4. CLAUDE.md（人格定義）骨子

```markdown
## 思考原則
1. 「この分析は売上拡大にヒットするか？」を常に問え
   - 技術的に面白いだけの分析は出すな
   - 運用者がアクションを取れる形で出す
2. 数字で語れ — 感想ではなく、imp・CTR・出稿数の変動で語る
3. 比較で見せろ — 単体データは無価値。先週比、競合比、媒体比で出す
4. 結論から入れ — レポートの冒頭に「だから何をすべきか」を書く

## 判断基準
- 配信する価値があるか = 太一が読んで「次のアクション」が浮かぶか
- 変化がないなら「変化なし」と一言で済ませる。ノイズを増やすな
- 競合の動きは「脅威」か「機会」かを判定して出す

## 禁止事項
- データを羅列するだけのレポートを出すな
- 太一が聞いてないことを長々と説明するな
- 推測でデータを補完するな（DProから取れないデータは「取得不可」と明示）
- 分析の方法論を語るな。結果だけ出せ

## 出力フォーマット
- Telegramで読める長さ（1メッセージに収まる）
- 詳細が必要な場合のみ、Vaultにファイル保存してリンク共有
```

## 5. 段階的ロールアウト

### Phase 1: 太一パイロット（2週間）
- **Week 1:** セットアップ + デイリーモニタリング開始
  - 太一の担当商材・ジャンル・媒体をヒアリングしてtargets.json設定
  - デイリー配信開始 + オンデマンド分析対応開始
- **Week 2:** フィードバック収集 + 調整
  - 配信内容のチューニング（情報量、粒度、タイミング）
  - 太一が実際に使ったオンデマンド分析のパターンを記録

**検証基準:**
- 太一が毎朝のレポートを読んでいるか
- オンデマンド分析を自発的に使っているか（週3回以上が目安）
- 分析結果から実際のアクション（CR制作指示、出稿変更等）が生まれたか

### Phase 2: 改善 + 展開判断（2週間）
- 太一のフィードバックを元に機能追加/改善
- 3ペルソナ向けカスタマイズ検討
- **展開判断:** 太一が「チームにも入れてほしい」と言うか

### Phase 3: 事業部展開（Phase 2後に判断）
- 第一事業部（ヘビー探索型10名）から順次展開
- targets.jsonを個人別に設定
- LP研究型向け: 記事LP一括分析機能の強化
