# ダッシュボード要件定義 v2（改訂版）

> v1からの変更点: 質問①〜⑤の回答 + セルフレビューの穴5点を反映

## 概要

影分身（OpenClawエージェント）の活動をリアルタイムで可視化するWebアプリ。
にっしーが1人で10人の影分身を一元管理するためのツール。

---

## 前提（確定事項）

| 項目 | 決定 |
|------|------|
| MVPデプロイ先 | Mac miniローカル起動（localhost:3000） |
| Vercelデプロイ | 後回し。必要になったら |
| 影分身ごとのbot | 分ける（BotFather + OpenClawマルチエージェント） |
| タスク割り当て | 当面Telegram経由 |
| ドメイン | 不要（localhost） |
| UI方針 | MVP: 実用寄り（A）→ Phase 2: オフィス画面（C） |
| 認証 | Basic Auth（環境変数で設定） |
| DB | MVPでは使わない。ファイルベース運用 |

## データソースと取得方法（改訂）

### 1. エージェントステータス
| データ | 取得元 | 方法 |
|--------|--------|------|
| エージェント一覧 | OpenClaw Gateway | WS `system-presence` |
| セッション情報 | OpenClaw Gateway | WS `sessions.list`（30秒） |
| トークン使用量 | OpenClaw Gateway | WS `sessions.list` → store fields |

### 2. 作業ログ（v1の「思考ログ」を改名）
| データ | 取得元 | 方法 |
|--------|--------|------|
| 作業ログ | ワークスペース | `memory/YYYY-MM-DD.md` を読む（60秒） |
| Telegram送信履歴 | Gateway | `sessions.history`（30秒） |

> ⚠️ v1で `thinking_log/*.md` を参照していたが廃止。`memory/*.md` に統一。

### 3. タスク・質問
| データ | 取得元 | 方法 |
|--------|--------|------|
| タスク一覧 | 各リポジトリ | `tasks.json`（60秒） |
| 質問一覧 | 各リポジトリ | `questions/*.md`（60秒） |

### 4. コミット履歴
| データ | 取得元 | 方法 |
|--------|--------|------|
| コミット一覧 | GitHub API | `GET /repos/{owner}/{repo}/commits`（5分） |

### ファイル読み取りの方式
MVPはローカル起動なので、Next.js API Routeから直接ファイルシステムを読める:
```
[Browser] → [Next.js localhost:3000] → fs.readFile(ワークスペースパス)
                                     → Gateway WS (localhost:18789)
                                     → GitHub API (HTTPS)
```
Vercel移行時はGateway API経由に切り替える。

## リアルタイム更新方式（改訂）

### MVPはSWR pollingのみ

ローカル起動なので複雑なSSEは不要。SWRの `refreshInterval` で十分:
- ステータス: 30秒
- 作業ログ: 60秒
- タスク/質問: 60秒
- コミット: 5分

### Phase 2でSSE追加検討
Vercel移行時にSSEが必要になったら追加。

## 技術選定（改訂）

| レイヤー | 技術 | 理由 |
|----------|------|------|
| フレームワーク | Next.js 15 (App Router) | AXADと統一 |
| UI | React 19 + Tailwind CSS 4 + shadcn/ui | AXADと統一 |
| アニメーション | Framer Motion | ステータス変化のトランジション |
| データ取得 | SWR | AXADと統一。polling + キャッシュ |
| リアルタイム | SWR polling（MVP）→ SSE（Phase 2） | 段階的 |
| デプロイ | ローカル（MVP）→ Vercel（Phase 2） | 段階的 |
| 認証 | Basic Auth（環境変数） | MVPの最低限セキュリティ |
| DB | なし（ファイルベース） | 最速リリース |

## タスク管理の規約（改訂）

各影分身のリポジトリに `tasks.json`:

```json
{
  "tasks": [
    {
      "id": "task-001",
      "title": "AXAD Phase2 Step1 要件定義",
      "status": "in_progress",
      "priority": "high",
      "created": "2026-03-01T21:00:00+09:00",
      "updated": "2026-03-01T22:05:00+09:00",
      "subtasks": [
        { "title": "リサーチ", "status": "done" },
        { "title": "DBスキーマ設計", "status": "in_progress" },
        { "title": "API設計", "status": "todo" },
        { "title": "UIフロー", "status": "todo" }
      ]
    }
  ]
}
```

### 更新ルール（CLAUDE.mdに明記する）
- タスク開始時: statusを `in_progress` に、updatedを更新
- タスク完了時: statusを `done` に、updatedを更新
- ブロック時: statusを `blocked` に、理由をdescriptionに記載
- 新タスク受領時: tasks.jsonに追加

## 質問管理の規約

各影分身の `questions/*.md`:
```markdown
---
id: q-001
from: nishiyama001
to: nishiyama
status: open
created: 2026-03-01T22:00:00+09:00
---
# 質問タイトル
質問本文
```

### 回答フロー（MVP）
1. ダッシュボードで未回答質問を確認
2. **Telegram経由で影分身に回答**（既存フロー）
3. 影分身がquestions/*.mdのstatusを `answered` に更新

> Phase 2でダッシュボードから直接回答→影分身に通知の機能を追加検討
