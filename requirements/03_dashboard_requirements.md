# ダッシュボード要件定義

## 概要

影分身（OpenClawエージェント）の活動をリアルタイムで可視化するWebアプリ。
にっしーが1人で10人の影分身を一元管理するためのツール。

---

## ユーザーストーリー

### 必須（MVP）

| # | ストーリー | 優先度 |
|---|-----------|--------|
| US-1 | にっしーとして、全影分身の現在のステータス（稼働中/待機中/エラー）を一覧で見たい | 🔴 |
| US-2 | にっしーとして、各影分身が今どのタスクに取り組んでいるかリアルタイムで知りたい | 🔴 |

| US-5 | にっしーとして、各影分身のコミット履歴を時系列で確認したい | 🟠 |
| US-6 | にっしーとして、タスクの進捗（TODO→作業中→完了）をカンバンで管理したい | 🟠 |

### 将来（Phase 2以降）

## データソースと取得方法

### 1. エージェントステータス

| データ | 取得元 | 方法 | 更新頻度 |
|--------|--------|------|----------|
| エージェント一覧 | OpenClaw Gateway | WebSocket `system-presence` | リアルタイム |
| セッション情報 | OpenClaw Gateway | WebSocket `sessions.list` | 30秒polling |
| トークン使用量 | OpenClaw Gateway | WebSocket `sessions.list` → store fields | 30秒polling |

### 2. 思考ログ・アクティビティ

| データ | 取得元 | 方法 | 更新頻度 |
|--------|--------|------|----------|
| 最新の思考 | ワークスペースファイル | `memory/YYYY-MM-DD.md` を読む | 60秒polling |
| 作業ログ | ワークスペースファイル | `thinking_log/*.md` を読む | 60秒polling |
| Telegramメッセージ | セッション履歴 | `sessions.history` | 30秒polling |

### 3. タスク管理

| データ | 取得元 | 方法 | 更新頻度 |
|--------|--------|------|----------|
| タスク一覧 | 各リポジトリ | `tasks.json` or GitHub Issues | 60秒polling |
| 質問一覧 | 各リポジトリ | `questions/*.md` を読む | 60秒polling |

### 4. コミット履歴

| データ | 取得元 | 方法 | 更新頻度 |
|--------|--------|------|----------|
| コミット一覧 | GitHub API | `GET /repos/{owner}/{repo}/commits` | 5分polling |
| ファイル変更 | GitHub API | `GET /repos/{owner}/{repo}/commits/{sha}` | オンデマンド |

## リアルタイム更新方式

### 推奨: SSE (Server-Sent Events) + SWR

```
[Browser] ←SSE── [Next.js API Route] ←WS── [OpenClaw Gateway]
                                      ←HTTP─ [GitHub API]
                                      ←FS──  [Workspace Files]
```

**理由:**
- WebSocketはクライアント↔サーバー間では過剰（単方向で十分）
- SSEはNext.jsのAPI Routeで簡単に実装可能
- SWRのrevalidateとSSEを組み合わせて最新データを維持
- Gateway側はWebSocket接続を1本維持し、Next.jsバックエンドで集約

### フォールバック: Polling

SSEが不安定な場合はSWRのrefreshIntervalで30秒pollingにフォールバック。

## 技術選定

| レイヤー | 技術 | 理由 |
|----------|------|------|
| フレームワーク | Next.js 15 (App Router) | Phase 1と統一。SSR + API Routes |
| UI | React 19 + Tailwind CSS 4 + shadcn/ui | Phase 1と統一 |
| アニメーション | Framer Motion | ゲーミフィケーション的なUI演出 |
| データ取得 | SWR | Phase 1と統一。キャッシュ+リアルタイム |
| リアルタイム | SSE + SWR revalidate | 軽量なリアルタイム |
| チャート | Recharts | Phase 1と統一 |
| デプロイ | Vercel Pro | Phase 1と統一 |
| DB | Neon PostgreSQL（オプション） | タスク管理に必要な場合のみ |

### DBを使うか？

**Phase 1（MVP）: DBなし**
- タスク管理は各リポジトリの `tasks.json` で管理
- エージェント情報はOpenClaw Gateway APIから取得
- 質問管理は各リポジトリの `questions/*.md` で管理
- → 追加インフラなしで最速リリース

**Phase 2以降: DB追加検討**
- タスク管理が複雑化したら
- コスト履歴の蓄積が必要になったら
- エージェント間の依存関係管理が必要になったら

## タスク管理の規約

各影分身のリポジトリに `tasks.json` を置く:

```json
{
  "tasks": [
    {
      "id": "task-001",
      "title": "AXAD Phase2 Step1 要件定義",
      "status": "in_progress",
      "priority": "high",
      "created": "2026-03-01T21:00:00+09:00",
      "updated": "2026-03-01T22:05:00+09:00",
      "description": "記事一覧DB化の機能要件定義",
      "subtasks": [
        { "title": "リサーチ", "status": "done" },
        { "title": "DBスキーマ設計", "status": "in_progress" },
        { "title": "API設計", "status": "todo" },
        { "title": "UIフロー", "status": "todo" }
      ]
    }
  ]
}
```

ステータス: `todo` | `in_progress` | `done` | `blocked`

## 質問管理の規約

各影分身の `questions/` ディレクトリに質問ファイルを置く:

```markdown
---
id: q-001
from: nishiyama001
to: nishiyama（にっしー）
status: open
created: 2026-03-01T22:00:00+09:00
---

# 記事一覧スプレッドシートのURL

Step 1の設計に入るにあたり、記事一覧の現状のデータ構造を把握したい。
スプレッドシートのURLと、主要な列の説明を教えてほしい。
```

ステータス: `open` | `answered` | `closed`
