# nishiyama002 起動方式比較 — OpenClaw vs Codex CLI

_作成: 2026-03-05 10:45 by 1号_
_目的: にっしーが即判断できるよう、両方式のメリデメを整理_

---

## 結論（1号推奨）

**OpenClawエージェント方式を推奨。** 理由: 1号・5号との連携、heartbeatによる自律保守、Telegram経由の指示がシームレスにできる。Codex CLIは「呼んだときだけ動く」ので、2号に期待する常駐型の保守運用には向かない。

---

## 比較表

| 観点 | OpenClaw エージェント | Codex CLI |
|---|---|---|
| **起動** | 常駐（heartbeatで自律稼働） | 手動（コマンド実行時のみ） |
| **1号→2号 連携** | sessions_send で即指示 | exec でCLI起動 → 完了待ち |
| **にっしー→2号 指示** | Telegram から直接 | ターミナルで codex コマンド |
| **5号（品質管理）連携** | sessions_send / subagents | 不可（独立プロセス） |
| **自律保守** | heartbeatで定期チェック可能 | 不可（呼ばないと動かない） |
| **コスト** | heartbeat分のトークン消費あり | 使った分だけ |
| **セットアップ工数** | agent create + 設定（30分程度） | CLI確認のみ（10分） |
| **向いてるユースケース** | 常駐保守、自律バグ検知、チーム連携 | 単発タスク、スポット開発 |

## 1号の見解

2号の役割は「AXAD専属の開発・保守」。Phase 2開発が始まれば継続的にコードを書くし、Phase 1の保守も並行する。これはCodex CLIの「呼んだら来る助っ人」ではなく、OpenClawの「チームメンバー」として動かすべき。

**ハイブリッド案もあり:** OpenClawで常駐しつつ、重い実装タスクはCodex CLIをサブプロセスとして呼ぶ。1号が3号にsessions_spawnでコーディングを振るのと同じパターン。

## 判断に必要な情報

にっしーに聞きたいのは1点:
- **2号を常駐させたいか、呼んだときだけ動けばいいか？**

常駐 → OpenClaw。スポット → Codex CLI。それだけ。
