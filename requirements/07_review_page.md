# 自動精査ページ改修 設計書

## 1. 背景と課題

### 現状
- 精査ルールはスプレッドシートの巨大なARRAYFORMULA（IFネスト100+段）で管理
- Metaだけで80件以上、Google/TikTok含め200件超のルールセット
- にっしー1人が全ルールを管理（完全属人化）
- AXADにルール設定ページはあるがmaster限定＋エンジニア向けUI

### ゴール
- **各部長・課長が自分で精査基準を編集できる**
- **UIが直感的で操作しやすい**（正規表現やJSONを見せない）
- **導入後はにっしーに手戻りがない**

---

## 2. ユーザーと権限

### アクセス制御

| ロール | 閲覧範囲 | 編集範囲 |
|--------|---------|---------|
| master | 全ルール | 全ルール |
| ad_admin | 全ルール | 全ルール |
| director（本部長） | 全ルール | 全ルール |
| manager（部長） | 全ルール | 自事業部のルールのみ |
| leader（課長） | 全ルール | 自課のルールのみ |
| member | 閲覧不可 | 編集不可 |

- ルールの「所有者」はチーム/担当者パターンで判定
- 自分が編集できないルールはグレーアウト表示（閲覧はできる）

## 3. UIデザイン

### 3.1 ページ構成

```
/dashboard/rules（精査ルール設定）
  ├── ルール一覧（優先度順）
  ├── ルール作成/編集フォーム
  ├── 競合チェック結果パネル
  └── シミュレーションパネル
```

### 3.2 ルール一覧

- **優先度順（自動計算）で表示**
- 各ルールカードに表示する情報:
  - ルール名（例：「TM2 × TCB-二重女性」）
  - 適用条件サマリー（媒体、チーム、担当者、案件）
  - 停止閾値の概要（例：「CV=0: ¥2,000 / CV≧1: ROAS 1.2未満」）
  - 有効/無効トグル
  - 適用CP数（現在何件のCPに影響してるか）
  - ⚠️ 警告バッジ（競合あり / 対象0件 / デッドルール）
- **フィルタ**: 媒体 / チーム / 担当者 / 案件 / 有効/無効
- 自分が編集できないルールはグレーアウト

### 3.3 ルール作成 — 3つの入口

#### A. テンプレートから作る
1. テンプレートを選択（プリセット or カスタム保存済み）
2. フィルタ条件を設定（媒体、チーム、担当者、案件）
3. 数字がプリフィルされた状態で表示
4. 必要に応じて微調整
5. 保存

#### B. 既存ルールを複製
1. 一覧から既存ルールを選んで「複製」
2. フィルタ条件と数字を編集
3. 保存

#### C. ゼロから作る
1. フィルタ条件を設定
2. 停止条件を1つずつ追加
3. 保存

### 3.4 ルール編集フォーム

#### フィルタ条件（どのCPに適用するか）

全てドロップダウン or テキスト入力。正規表現は内部変換で隠蔽する。

| 項目 | 入力方式 | 例 |
|------|---------|-----|
| 媒体 | ドロップダウン（Meta/TikTok/Google/YouTube/X/ALL） | Meta |
| チーム | ドロップダウン（DBから取得） | TM2 |
| 担当者 | ドロップダウン（DBから取得） | 白井 |
| 案件 | ドロップダウン or テキスト入力 | TCB-二重女性 |
| 予算帯 | 数値入力（下限〜上限） | ¥50,000〜¥100,000 |
| ステータス | ドロップダウン（ON/OFF/ALL） | ON |

- 複数選択可能（例：チーム = TM1-1 or TM1-2）
- 「案件」は既存キャンペーン名からサジェスト

#### 停止条件テーブル（CV段階別）

```
┌──────────────────────────────────────────────────┐
│ 停止条件                                          │
├──────────┬───────────────────────────────────────┤
│ CV = 0   │                                       │
│  MCV = 0 │ Cost > [    ] 円で停止                │
│  MCV > 0 │ Cost > [    ] 円 かつ CPC > [    ] 円 │
│  上限    │ Cost > [    ] 円で停止                │
├──────────┼───────────────────────────────────────┤
│ CV = 1   │ Cost > [    ] 円 かつ MCPA > [    ] 円│
├──────────┼───────────────────────────────────────┤
│ CV ≧ 1   │ ROAS < [    ] で停止                  │
│ CV ≧ 2   │ ROAS < [    ] で停止                  │
│ CV ≧ 3   │ ROAS < [    ] で停止                  │
│ CV ≧ 5   │ ROAS < [    ] で停止                  │
│ CV ≧ 7   │ ROAS < [    ] で停止                  │
├──────────┼───────────────────────────────────────┤
│ + 条件追加                                        │
└──────────────────────────────────────────────────┘
```

- 各行は有効/無効をトグルできる（不要な段階はOFF）
- 「+ 条件追加」でカスタム条件も追加可能（フィールド + 演算子 + 値）
- **数字を入力するだけ。正規表現やJSON条件式は一切見えない**

#### 再開ルール（OFF CP用）

```
┌──────────────────────────────────────────────────┐
│ 再開条件（停止中CPの再開判定）                     │
├──────────────────────────────────────────────────┤
│ ROAS > [    ] かつ MCPA < [    ] 円 → 再開       │
└──────────────────────────────────────────────────┘
```

#### グローバルルール

```
┌──────────────────────────────────────────────────┐
│ グローバル停止条件（全CPに適用、最優先）           │
├──────────────────────────────────────────────────┤
│ 利益 < -[50,000] 円 → 即時停止                   │
│ 利益 < -[20,000] 円（担当者: [悠太]）→ 即時停止  │
└──────────────────────────────────────────────────┘
```

## 4. 優先度の自動計算

### 具体性スコア

ルールの条件が具体的なほど優先度が高い。

| 条件 | スコア |
|------|--------|
| 媒体指定（非ALL） | +10 |
| チーム指定 | +20 |
| 担当者指定 | +30 |
| 案件指定 | +40 |
| 予算帯指定 | +5 |

**合計スコアが高いルールが優先。** 同スコアなら後から作成されたルールが優先。

例：
- 「全チーム × 全案件」 → スコア 10（媒体のみ）
- 「TM2 × TCB-二重」 → スコア 70（媒体+チーム+案件）
- 「TM2 × 白井 × TCB-二重」 → スコア 100（媒体+チーム+担当者+案件）

### 一覧表示

優先度の高いルールが上。同じCPに複数ルールがマッチした場合、最上位のルールだけが適用される。

## 5. 競合チェック

### 保存時の自動チェック

新ルール保存時に以下を自動検出：

1. **対象CP重複**
   - 新ルールと既存ルールの対象CPが重なる場合に警告
   - 「このルールは既存ルール「TM2 × TCB-二重」と対象が重複しています」
   - どちらが優先されるかを明示

2. **完全包含（デッドルール）**
   - 上位ルールに完全に覆われて適用されないルールを検出
   - 「このルールは「TM2 × 白井 × TCB-二重」が常に優先されるため、白井さんのCPには適用されません」

3. **対象0件**
   - 「この条件に該当するキャンペーンが現在0件です」
   - ※ 将来追加される可能性があるので警告のみ（保存は可能）

### 警告の表示

- ⚠️ 黄色バッジ: 軽微な競合（対象重複あるが意図通りの可能性）
- 🔴 赤バッジ: デッドルール（絶対に適用されない）
- ℹ️ 青バッジ: 対象0件（情報提供）

## 6. シミュレーション機能

### 保存前プレビュー

「保存」ボタンを押す前に「シミュレーション」ボタンで影響を確認：

```
┌──────────────────────────────────────────────────┐
│ シミュレーション結果                               │
├──────────────────────────────────────────────────┤
│ 対象CP数: 42件                                    │
│                                                    │
│ 判定変更:                                          │
│   継続 → 停止: 3件                                │
│   停止 → 継続: 0件                                │
│   手動運用 → 停止: 1件                            │
│                                                    │
│ 変更対象の詳細:                                    │
│   ├ TM2_白井_TCB-二重女性_...  継続→停止          │
│   │  理由: Cost ¥3,200 > ¥3,000（CV=0, CPC=¥550）│
│   ├ TM2_白井_TCB-二重女性_...  継続→停止          │
│   │  理由: ROAS 1.15 < 1.2（CV=2）               │
│   └ ...                                           │
│                                                    │
│ [キャンセル]              [この内容で保存]         │
└──────────────────────────────────────────────────┘
```

- 現在の当日CPデータに対してルールを適用した結果を表示
- 「判定が変わるCP」だけをハイライト
- 影響が大きい場合（10件以上変更）は追加確認

## 7. テンプレート

### プリセットテンプレート

| テンプレート名 | 用途 | CV=0上限 | CV≧1 ROAS | CV≧5 ROAS |
|---------------|------|---------|-----------|-----------|
| 標準 | 大半の案件 | ¥5,000 | 1.3 | 1.6 |
| 高単価案件 | 単価が高い案件 | ¥8,000 | 1.5 | 1.7 |
| 低単価案件 | 単価が低い案件 | ¥3,000 | 1.2 | 1.5 |
| 新規テスト | 立ち上げ期 | ¥10,000 | 1.0 | 1.3 |
| 検証用（SC/見出し） | TM2の検証系 | ¥6,000 | 1.2 | 1.5 |

### カスタムテンプレート

- 部長・課長が自分のチーム用テンプレートを保存可能
- 「このルールをテンプレートとして保存」ボタン
- 新案件追加時にテンプレートから一発で作成

## 8. DBスキーマ変更

### 既存テーブル: rules

変更なし。既存のconditions JSON + フィルタフィールドで対応可能。

### 新規テーブル: rule_templates

```sql
CREATE TABLE rule_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  is_preset BOOLEAN DEFAULT false,  -- プリセット or カスタム
  created_by UUID REFERENCES users(id),
  team VARCHAR(100),  -- カスタムテンプレートの所有チーム
  conditions JSON NOT NULL,  -- テンプレートの条件（数値のみ可変）
  filter_defaults JSON,  -- フィルタのデフォルト値
  description VARCHAR(500),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

### 既存テーブル: rules に追加カラム

```sql
ALTER TABLE rules ADD COLUMN specificity_score INTEGER DEFAULT 0;
ALTER TABLE rules ADD COLUMN template_id UUID REFERENCES rule_templates(id);
ALTER TABLE rules ADD COLUMN owner_team VARCHAR(100);  -- 編集権限判定用
ALTER TABLE rules ADD COLUMN owner_section VARCHAR(100);
```

## 9. API仕様

### 既存API改修

#### GET /api/admin/rules
- `master`限定を撤廃 → `ad_staff`以上がアクセス可能
- レスポンスに`canEdit`フラグ追加（ユーザーの所属チームで判定）
- `specificity_score`順でソート

#### POST /api/admin/rules
- `master`限定を撤廃 → チーム所属チェックに変更
- 保存前に競合チェックを実行、結果をレスポンスに含む
- `specificity_score`を自動計算して保存

#### PATCH /api/admin/rules/[id]
- 編集権限チェック追加（自チームのルールのみ）

### 新規API

#### POST /api/rules/simulate
- リクエスト: ルール条件（保存前の仮ルール）
- レスポンス: 現在の当日CPに対する判定結果の差分
  - 変更件数、変更内容の詳細

#### GET /api/rules/conflicts?ruleId=xxx
- 指定ルールと競合する既存ルールを返す
- デッドルール判定、対象CP数も返す

#### GET /api/rules/templates
- テンプレート一覧（プリセット + ユーザーのカスタム）

#### POST /api/rules/templates
- カスタムテンプレート保存

#### GET /api/rules/filter-options
- フィルタ条件のドロップダウン候補を返す
- チーム一覧、担当者一覧、案件一覧（現在のCPデータから動的生成）

## 10. 既存ルール移行

### スプレッドシート → DB移行手順

1. スプレッドシートの全ルールを解析（にっしーが確認）
2. チーム × 担当者 × 案件の組み合わせごとにルールレコードを作成
3. 各ルールのCV段階別閾値をconditions JSONに変換
4. specificity_scoreを自動計算
5. **移行後にシミュレーションを実行して、スプレッドシートと同じ判定結果になることを検証**

### 移行スクリプト

`scripts/migrate-review-rules.ts` として作成。にっしーが実行して検証。
